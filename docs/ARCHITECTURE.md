## Project Description

A mobile app for home workouts with game-like progression mechanics.
The user progresses from basic exercises (knee push-ups) to advanced
calisthenics (handstand push-ups) through short daily sets of 5‚Äì15 minutes.

**Principles:** all data is local, no backend, free, no ads.

## Key Concepts

### Set (Set / Workout)
A short session: 3‚Äì6 exercises, 5‚Äì15 minutes. Types: `Daily`, `Skill`, `Challenge`.

### Progression Branches
5 branches: Push, Pull, Core, Legs, Balance. Each is a progression ladder from simple to advanced.
Pull requires a pull-up bar (`requiresEquipment = true`). `UserProfile.activeBranches` is a computed getter
that filters out Pull if `hasPullUpBar == false`.

### In-Stage Progression
Reps ‚Üë ‚Üí Sets ‚Üë (with reps reset) ‚Üí Rest ‚Üì ‚Üí Challenge test ‚Üí Next stage.

### Gamification
- **SP (Strength Points)** ‚Äî points earned for exercises
- **Streak** ‚Äî consecutive days; freezes (max 3, earned every 7 streak days)
- **Ranks:** Beginner ‚Üí Amateur ‚Üí Athlete ‚Üí Champion ‚Üí Master ‚Üí Legend
- **Achievements** ‚Äî 27 total, checked after each workout and stage advance
- **Bonus workouts** ‚Äî multiple workouts per day are allowed (50% SP, progression does not advance)

### Primary vs Bonus Workout
`WorkoutLog.isPrimary`: true = first of the day (full SP, advances progression); false = 50% SP.
Determined at start: if a `WorkoutLog` already exists for today ‚Üí `isPrimary = false`.

## Services (Domain Layer)

### SPService
- `forExercise(result, exercise)` ‚Äî SP for an exercise; warmup/cooldown (stage=0) ‚Üí 0
- `forWorkout(...)` ‚Äî totals SP; primary = √ó1.5 (daily bonus) √ó 1.1 (completion); bonus = √ó0.5
- `applyToProfile(profile, sp)` ‚Äî mutates profile: SP + rank

### StreakService
- `applyWorkout(profile, date)` ‚Üí bool ‚Äî updates streak; on a 1-day gap, uses a freeze
- `tryAwardFreeze(profile)` ‚Üí bool ‚Äî +1 freeze every 7 days (max 3)
- `isStreakAtRisk(profile)` ‚Äî true if no workout today
- `daysSinceLastWorkout(profile)` ‚Äî counter

**displayStreak (important):** computed on the fly in `homeDataProvider` without mutating Hive:
```
days <= 1                          ‚Üí profile.currentStreak
days == 2 && freezeCount > 0       ‚Üí profile.currentStreak  (freeze will save it on next workout)
otherwise                          ‚Üí 0
```

### ProgressionService
- `applyResult(progress, exercise, result)` ‚Üí bool (true = challenge unlocked)
- `advanceStage(progress, nextExercise)` ‚Äî stage transition after challenge
- `applyRegression(progress, exercise, daysSkipped)` ‚Äî rollback on 3+ days skipped
- `nextExercise(progress)` ‚Äî next stage from catalog

### WorkoutGeneratorService
- `generateDaily({activeBranches, preferredMinutes, dayIndexOverride?})` ‚Äî rotates branches by dayIndex (days since 2020-01-01). N branches: min(2,total) at ‚â§5min, min(3,total) at 10min, total at ‚â•15min
- `generateChallenge(branch)` ‚Äî warmup ‚Üí current stage (1 easy set) ‚Üí next stage (challengeTargetReps) ‚Üí cooldown

### AchievementService
- `checkAfterWorkout({profile, log, totalWorkouts, achievementRepo})` ‚Üí `List<String>` new ones
- `checkAfterStageAdvance({branch, newStage, allProgress, achievementRepo})` ‚Üí `List<String>` new ones

### SoundService (singleton)
- Methods: `tick()`, `ding()`, `pop()`, `complete()`
- **Do NOT use `await player.stop()` before play()** ‚Äî causes Android MediaPlayer reset (~100‚Äì200ms). `player.play()` handles restart on its own.
- Configured from `SettingsNotifier` on load and on settings change

### Mutation Pattern
All services **mutate** the passed objects (`UserProfile`, `SkillProgress`) in place.
Persistence is the responsibility of the calling code via repositories.

## Health Integration

- **Write:** workout session (`TRADITIONAL_STRENGTH_TRAINING`) + calories (MET 5.5 √ó 70kg √ó hours)
- **Read:** body weight (optional, for more accurate calorie calculation)
- All features are opt-in via Settings ‚Üí HEALTH
- `HealthService.instance.configure()` is wrapped in `.timeout(5s)` (otherwise hangs on splash)

**Critical for Android:**
- `MainActivity` extends `FlutterFragmentActivity` (not `FlutterActivity`)
- `activity-alias` with `HEALTH_PERMISSIONS` in AndroidManifest is required for the permissions dialog

**iOS:** `Runner.entitlements` contains `com.apple.developer.healthkit` + App Group. `CODE_SIGN_ENTITLEMENTS` is set in all 3 build configs in `project.pbxproj`. HealthKit capability must also be enabled in Xcode (Runner ‚Üí Signing & Capabilities ‚Üí + ‚Üí HealthKit) for distribution builds.

## Android Specifics

- `minSdk = 26` (health package requires API 26+)

- Small icon (monochrome, status bar): `AndroidInitializationSettings('ic_goro_notif')` + `drawable/ic_goro_notif.xml`

- `updateWidget(qualifiedAndroidName: ...)` ‚Äî use `qualifiedAndroidName`, not `androidName` (otherwise the package appends the package name twice)

## Code Style and Conventions

- **Commit messages ‚Äî always in English**
- Dart ‚Äî official Dart style guide
- Widgets ‚Äî prefer `StatelessWidget` + Riverpod `ConsumerWidget`
- File naming: `snake_case`
- Class naming: `PascalCase`
- Public API comments in English
- UI strings via l10n (English primary, Russian secondary; `app_en.arb` is the template)
- No clutter: do not add docstrings/comments to code you are not touching

### Key Patterns

- `homeDataProvider = Provider.autoDispose` + `_ref.invalidate(homeDataProvider)` after workout
- `workoutProvider = StateNotifierProvider.autoDispose` ‚Äî plan is generated on screen open
- Services mutate HiveObjects in place; persistence is the responsibility of the caller
- `ExerciseResult.completedReps` = rep count from the **last** set (MVP)
- Challenge in a bonus workout: `advanceStage` is called ALWAYS (regardless of `isPrimary`)
- Deep link guard: `if (state.uri.scheme == 'caliday') return '/workout';` in RouterNotifier redirect
- Workout ‚Üí check `canPop()` before `pop()`: `context.canPop() ? context.pop() : context.go('/home')`
